01 Foundation

The Stack

Every tool here is something I ship real products with. When I say "use this," it's because I've tried the alternatives and made the call. Opinionated by design.

Next.js (App Router)
Use

Full-stack React framework. Server components, API routes, file-based routing, and Vercel deployment that just works. The App Router (not Pages) is the right default in 2026 — it handles server/client rendering decisions cleanly and the mental model is sound once you understand it.

For a solo dev, the tight Vercel integration alone saves hours per month. No DevOps. No Docker. Push to main and it ships. TypeScript support is first-class. The ecosystem around it — Supabase, Resend, Stripe — all have Next.js examples.
TypeScript
Use

Types everywhere, no exceptions. The upfront overhead pays back in 30 minutes on the first refactor. AI-generated code with TypeScript is dramatically easier to review than untyped JavaScript — you can see exactly what shape a function expects and returns.

30 years of debugging production systems with no type safety. TypeScript catches 40% of my bugs before they run. With AI generating large chunks of code, the type checker is the second reviewer. Non-negotiable.
Supabase
Use

Postgres database, authentication, storage, and edge functions — all in one platform. RLS (Row Level Security) policies built into the database layer. The free tier is genuinely useful in production. Self-hostable when you outgrow the managed service.

PlanetScale is MySQL and serverless connections add latency. Firebase is a NoSQL trap that makes reporting hard and migrations painful. Neon is good but Supabase's auth integration alone saves a week of work. Postgres is the right database for 95% of apps — use it.
Vercel
Use

Deployment, CDN, edge functions, preview environments on every pull request. The free tier handles real production traffic. Scales without configuration. Next.js and Vercel are built by the same team — the integration is frictionless by design.

Railway is cheaper at scale but requires more configuration. Render is fine but slower cold starts. At solo dev stage, the hours you'd spend configuring Railway are worth more than the $20/month difference. Use Vercel until you have a reason to leave.
Claude API (Anthropic)
Use

The AI layer for anything requiring reasoning, generation, or analysis inside your product. Claude Sonnet is the right default — it handles complex instructions reliably and the context window is large enough for most product use cases without going to Opus.

I've shipped with both OpenAI and Claude. Claude follows complex system prompts more reliably — critical when you're building structured outputs or multi-step workflows. The tool use (function calling) implementation is cleaner. QurioLens runs on Claude API. It works.
Stripe
Use

Payments, subscriptions, and webhooks. The only serious option for a product charging real money. The dashboard, the test mode, the webhook CLI, the documentation — all first-class. The 2.9% + 30¢ fee is worth paying for everything you don't have to build.

I've seen developers try to save 0.5% by using LemonSqueezy or Paddle. The integration friction is real and the features you lose matter at early stage. Use Stripe until you're doing enough volume that the fee difference is significant. You won't be for a while.
Resend
Use

Transactional email — welcome emails, receipts, password resets, anything triggered by an API call. Clean SDK, React Email support for building HTML emails in JSX, excellent deliverability, and a free tier that handles real usage.

SendGrid is overkill and the UX is from 2014. Postmark is good but more expensive. Resend was built for developers and it shows — the API is clean, the documentation is honest, and React Email makes building HTML emails not terrible.
Tailwind CSS + CSS Modules
Use

Tailwind for structural utilities (layout, spacing, flex, grid). CSS Modules for anything that defines visual identity — custom gradients, animations, branded colors, hover effects. The hybrid keeps scaffolding fast and keeps the site from looking like a Tailwind template.

Pure Tailwind produces readable layout code but ugly, template-feeling sites. Pure CSS Modules is slow for scaffolding. The split — Tailwind for structure, CSS Modules for identity — is the right tradeoff for a product that needs to look distinct, not generic.
On switching costs
The instinct to evaluate every tool before committing is how developers spend six weeks on stack selection and zero weeks shipping. Pick this stack, learn it deeply, and ship. Migrate when a tool genuinely fails you — not because you read that something else is technically superior.
02
The Build Process
Workflow

Five stages from idea to revenue. The mistake most solo developers make is treating this as a linear checklist. It isn't — but each stage has a clear goal and a clear failure mode.

Stage 1 Ideate Goal: find a problem worth solving
What to do

Look for problems you've personally experienced or directly observed. Read 1-star reviews of tools in your domain. Spend time in communities where your target user complains publicly. Write down the problem in one sentence from the user's perspective — not yours.

Tools that help

Reddit (r/SideProject, r/SaaS, niche subreddits), Product Hunt reviews, App Store reviews, YouTube comments on competitors, direct conversations with 3–5 potential users.

You're building something because the technology is interesting, not because real people have a real problem. Technology-first products rarely find customers.
Stage 2 Validate Goal: prove demand before writing code
What to do

Answer 28 questions about the problem, the buyer, the competition, and your distribution path. Run the process manually for 3–5 people before building anything. Confirm that someone will pay before building the thing they'd pay for.

Tools that help

The Idea Validation Checklist (companion to this guide). A landing page with a waitlist. A Google Form that does the manual version of what you're about to automate. Direct outreach — not a survey, a conversation.

Skipping validation because you're "pretty sure" people will pay. The confidence of a developer who has never sold anything is a liability, not an asset.
Stage 3 Build Goal: ship the smallest version that proves the core assumption
What to do

Define the MVP as one paragraph — what it does, what it doesn't do, and what assumption it tests. Build only that. Use Cursor for implementation with the stack above. Every feature that isn't in the one paragraph gets cut until after launch.

Tools that help

Cursor (AI code editor — see Section 3). The full stack from Section 1. Framer Motion for UI polish that matters. React Hook Form for any user input. Radix UI for accessible interactive patterns.

Adding features because they're easy to build. Easy-to-build features that nobody asked for are the primary cause of delayed launches and abandoned projects.
Stage 4 Ship Goal: get real users on the thing
What to do

Run the shipping checklist in Section 5 before calling it live. Deploy to Vercel. Set up the Stripe webhook. Confirm the Resend emails fire. Set your success metric (number of paying users in 30 days). Then tell people directly — communities, DMs, the channel.

Tools that help

Vercel deploy previews. Stripe test mode. Supabase logs. Resend dashboard. PostHog or Plausible for analytics — lightweight, privacy-respecting, enough signal at early stage.

Waiting until it's perfect. It won't be perfect after launch either. Ship when it solves the problem — not when every edge case is handled.
Stage 5 Get Customers Goal: acquire the first 10 paying users
What to do

The first 10 customers come from direct outreach — not SEO, not ads, not Product Hunt. Find the 10 people who have the problem you solved and tell them it exists. YouTube and content distribution come after proof-of-concept, not before.

Tools that help

The communities you identified in Stage 1. Your existing email list (this is why you build it before the product). Your YouTube channel if it has traction. LinkedIn for B2B. Direct DMs to people who've publicly described the problem.

Spending Stage 5 energy on building new features instead of finding users. The product is already built. The only job now is distribution.
03
AI as a Build Tool
Leverage

Not as a toy. Not as a replacement. As a force multiplier for a developer who already knows what they're doing. The gap between "I tried AI" and "I build with AI professionally" is enormous — here's what's actually in that gap.

Tool Use it for Don't use it for
Cursor Active development — writing components, refactoring, adding features to an existing codebase. Best when the codebase context is loaded and Cursor understands your patterns. Greenfield architecture decisions. Let Claude think through the structure first, then build in Cursor.
Claude (chat) Architecture decisions, debugging strange errors, reviewing generated code for logic problems, designing database schemas, planning API structures before writing them. Boilerplate code generation in an active project — Cursor does this better with full file context.
Claude API The AI layer inside your product — generation, analysis, classification, extraction. Any user-facing AI feature that needs to be reliable, structured, and configurable. Simple text matching or lookups that don't need reasoning. An API call costs money every time — don't use it where a regex would do.

How I structure a Cursor session for full-stack work: CLAUDE.md in the project root is the single context document — it tells Cursor the stack, the file structure, the naming conventions, and the rules. Every Cursor prompt starts with "Read CLAUDE.md before starting." Without that anchor, Cursor makes assumptions that drift from your actual codebase within three sessions.

AI for generation vs AI for review: Generation is fast and usually good enough for boilerplate. Review is where the real value is. When Cursor generates a 200-line component, the useful question is not "does this work?" — it's "is there a logic error in the state management?" and "are the RLS policies this relies on actually set up?" That question requires a developer who understands the system. AI can help find the answer. It can't ask the question on its own.

What AI is genuinely bad at: Cross-file consistency. If you refactor a type in one file, Cursor will not automatically know that three other files depend on that type in a way that now breaks. TypeScript catches some of this. Code review catches the rest. AI doesn't. This is the senior developer advantage — pattern recognition across a whole system that AI still lacks.

The real multiplier
A developer with 5 years of experience using AI builds faster than they did without it. A developer with 30 years of experience using AI builds faster than a team of 5 without it. The multiplier scales with experience because experience is what AI doesn't have — knowing what to build, what to cut, and what will break in production.
04
Solo Dev Economics
Business

AI changes the unit economics of software development in ways that most developers haven't fully internalized yet. Here's what actually shifts when one person can build what used to take a team.

Time to MVP
2–6 weeks
With the stack above and Cursor, a focused solo developer can ship a working SaaS MVP in 2–6 weeks. Three years ago this was 3–6 months minimum.
Infrastructure cost at launch
~$0–20/mo
Vercel free tier, Supabase free tier, Resend free tier, Sender.net free tier. You can run a real product in production for near zero until you have paying users.
Break-even subscriber count
5–15 users
At $20–49/month, covering your infrastructure and AI API costs requires fewer than 15 subscribers. The margin economics of solo SaaS are unlike anything that existed before.
The dangerous assumption
Build = Revenue
AI makes building easier but does nothing for distribution. The bottleneck has shifted entirely to customer acquisition. Build less. Ship sooner. Spend more time on distribution than on features.

When to build vs buy vs skip: Build when it's core to your product — the thing that makes it different. Buy when it's infrastructure — payments, auth, email. Skip when it's a nice-to-have — anything a user didn't specifically ask for doesn't exist yet.

Scope discipline is the hardest skill: AI makes adding features trivially easy. That's the danger. Every feature you add is a feature you have to maintain, explain, and support. A product with 3 features done well converts better than a product with 15 features done adequately. The AI-assisted developer's biggest risk isn't moving too slow — it's building too much of the wrong thing.

Pricing at solo dev scale: Charge more than you think is reasonable. The value of saving a user 2 hours per week is not $7/month — it's $40. Developers chronically underprice because they're thinking about their time to build, not the user's time saved. Price on value delivered, not on effort invested.

05
Shipping Checklist
Pre-launch

Twenty things to verify before you call something live. Every item on this list is something I've shipped without at least once. Learn from that.

Error states on every form
Empty submission, invalid email, network failure, 5xx — all handled with real user-facing messages. Not console.log.
UX
Loading states on every async action
Button shows "Saving..." and disables during submission. Users click twice when nothing happens. Prevent it.
UX
Mobile layout at 390px
Test on an actual phone, not just browser dev tools. Text readable, buttons tappable, no horizontal scroll.
Mobile
Supabase RLS policies active
Every table has row-level security enabled. Users can only read and write their own data. Test this with a second account before launch.
Security
Environment variables in Vercel
SUPABASE_URL, SUPABASE_ANON_KEY, RESEND_API_KEY, STRIPE_SECRET_KEY, ANTHROPIC_API_KEY — all set in Vercel production environment, not just local .env.
Config
Stripe webhook registered and verified
The webhook endpoint is registered in Stripe dashboard pointing to your production URL. Signature verification is on. Test with Stripe CLI before launch.
Payments
Transactional emails sending correctly
Welcome email, receipt email, any triggered email — send a real test to a real inbox. Check spam folder. Check that first_name personalization is populated.
Email
404 and error pages exist
not-found.tsx, error.tsx, and global-error.tsx — all present, branded, and linking back to the homepage. Users hit these. Don't leave them stranded.
UX
Console errors: zero
Open DevTools on every page before launch. Zero errors, zero unhandled promise rejections. A console error in production is a bug you haven't fixed yet.
Code
Claude API error handling
Rate limit errors, timeout errors, and malformed responses all handled gracefully. The API will fail eventually. The user should never see a raw API error.
AI
Legal pages present: privacy, terms, cookies
Required for any product that collects email addresses or processes payments. Linked in the footer. Updated to reflect your actual data practices.
Legal
Analytics installed
Plausible or PostHog — something that tells you which pages people visit and where they drop off. You need this data within 24 hours of launch, not two weeks later.
Analytics
Offline behavior tested
DevTools → Network → Offline. Submit a form. What happens? The user gets a clear error message, not a blank screen or a spinner that never resolves.
UX
Meta tags: title, description, og:image
When someone shares your link on Twitter or LinkedIn, it should show a proper preview. Takes 20 minutes and makes every share look professional.
SEO
Favicon present
The default Next.js favicon marks you as someone who didn't finish. 5 minutes to replace it with your actual mark.
Brand
Dark mode and light mode both tested
If you support both — and you should — check every page in both modes. Invisible text on dark backgrounds is a common last-minute discovery.
UI
Database backup confirmed
Supabase has daily backups on paid plans. On free, export your schema and any seed data before launch. Know where your data lives and how to get it back.
Infra
API rate limits understood
Claude API, Stripe, Resend — all have rate limits. Know what they are before launch. A flood of signups should not break your app.
Infra
Custom domain pointed at Vercel
Not yourapp.vercel.app. Your actual domain. DNS propagation takes up to 48 hours — do this before launch day, not on launch day.
Config
You've used the product yourself for a week
Not a demo session. Real use, from a real account, completing the real flow. If you can't do this — what are you launching?
Product
Pre-launch score
0 / 20 items verified
Check off items above to see your launch readiness.
